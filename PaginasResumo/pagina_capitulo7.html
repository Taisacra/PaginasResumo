Capítulo 7



Melhoria do processo de software




Introdução:

A engenharia de requisitos é crucial no desenvolvimento de software, onde os engenheiros e outros envolvidos trabalham juntos para entender as necessidades do cliente e definir os requisitos do sistema. Isso envolve várias etapas, desde a concepção até a validação dos requisitos. Os artefatos resultantes garantem um entendimento claro do problema a ser resolvido, e a revisão regular é essencial devido à possibilidade de mudanças nos requisitos ao longo do projeto.



Objetivo

A engenharia de requisitos é fundamental no desenvolvimento de software, e seu objetivo é estabelecer uma base sólida para a construção do projeto. Ela abrange diversas tarefas, como concepção, levantamento, elaboração, negociação, especificação, validação e gestão de requisitos. Essas tarefas são realizadas de forma iterativa, na medida em que a equipe do projeto e os envolvidos reúnem informações. A engenharia de requisitos constrói uma ponte entre o projeto e a construção, garantindo assim, que o software atenda às necessidades do cliente. É essencial adaptar as técnicas de engenharia de requisitos às necessidades específicas do processo, projeto, produto da equipe.



Elementos, Categorias e tipos de estruturas envolvidas no conceito:

Concepção

Nessa fase as necessidades de negócios são esclarecidas, levando a um entendimento básico do problema a ser desenvolvido. Essa comunicação é feita por todos os envolvidos, partes interessadas e engenheiros, de modo que o projeto possa ser iniciado.

Levantamento

Nessa etapa de levantamento é crucial para entender as metas de negócios de um sistema ou produto. As metas são objetivos de longo prazo, funcionais ou não funcionais, como confiabilidade, segurança e usabilidade. Elas ajudam a explicar requisitos, gerenciar conflitos e servem de base para elaboração, verificação, gerenciamento de conflitos, negociação e evolução dos requisitos. É importante a mobilização dos envolvidos para poder compartilhar suas metas honestamente e estabelecer um mecanismo de priorização. A agilidade na engenharia de requisitos é essencial para transferir ideias dos envolvidos para a equipe de software de forma harmoniosa e sem atrasos, pois novos requisitos podem surgir durante o desenvolvimento iterativo.

Elaboração

Nessa nova etapa os esforços são concentrados na criação de um modelo de requisitos já refinado para identificar os aspectos da função, comportamento e das informações do software. Durante a elaboração dos requisitos, são criados e refinados cenários do usuário, descrevendo como os usuários interagirão com o sistema. Os cenários são analisados e identificado classes de análise, entidades do domínio de negócio visíveis para o usuário. A elaboração dos requisitos envolve a definição dos atributos e serviços de cada classe, bem como a identificação das relações entre elas. É fundamental saber quando parar nesse processo, focando em estabelecer uma base sólida para o projeto sem se perder em detalhes desnecessários.

Negociação

Para evitar conflitos entre as partes envolvidas no projeto devido as necessidades de cada lado possuir divergências, é essencial conciliar essas diferenças por meio da negociação. Nesse processo, as partes envolvidas devem ordenar e discutir a prioridade de seus requisitos. Uma negociação eficaz resultará em um acordo mutuamente satisfatório. Utilizando uma abordagem iterativa que priorize os requisitos, avalie os custos e riscos envolvidos, e resolva conflitos internos, é possível eliminar, combinar ou modificar requisitos para atender a todas as partes envolvidas.

Especificação

A especificação no processo de construção do projeto pode assumir diferentes significados, podendo ser um documento escrito, um conjunto de modelos gráficos, um modelo matemático formal, um conjunto de casos de uso, um protótipo ou qualquer combinação dos fatores citados. No entanto, todos esses elementos têm o mesmo objetivo único: ser desenvolvidos e utilizados para especificar e argumentar de forma consistente e compreensível os requisitos apresentados.

Validação

Nessa fase de validação os artefatos produzidos são avaliados em relação à sua qualidade. A consistência é uma preocupação fundamental nesse processo. A validação de requisitos verifica se todos os requisitos de software foram declarados de forma clara e sem ambiguidades, detectando e corrigindo inconsistências, omissões e erros. Além disso, é verificado se os artefatos estão em conformidade com os padrões estabelecidos para o processo, projeto e produto. A revisão técnica é o principal mecanismo de validação de requisitos, envolvendo engenheiros de software, clientes, usuários e outros participantes para examinar a especificação em busca de erros, áreas que necessitam de esclarecimentos, informações faltantes, inconsistências e requisitos conflitantes ou inatingíveis.

Gerenciamento de requisitos

Ao longo da vida de um sistema, os requisitos tendem a mudar. A gestão de requisitos, que consiste em um conjunto de atividades, auxilia a equipe de projeto na identificação, controle e acompanhamento das necessidades e suas alterações à medida que o projeto avança. Essas práticas são semelhantes às técnicas de gerenciamento de configurações de software.



Estratégias e Metodologias:

Estabelecimento da base de trabalho

O estabelecimento da base de trabalho é um processo no qual os engenheiros de requisitos trabalham para estabelecer as bases necessárias para o entendimento dos requisitos de software, mesmo quando os clientes ou usuários estão em locais diferentes e distancias consideráveis, possuem conhecimento limitado ou opiniões conflitantes. Nesse cenário, é importante superar essas limitações para iniciar o projeto em direção a uma solução bem-sucedida. Os envolvidos (stakeholder) se refere a qualquer pessoa que se beneficia direta ou indiretamente do sistema em desenvolvimento. Os envolvidos incluem uma variedade de pessoas, como gerentes, clientes, usuários, engenheiros e outros, cada um com uma visão diferente do sistema, benefícios esperados com o sucesso do desenvolvimento e riscos associados ao fracasso do projeto. Devido à diversidade de envolvidos, os requisitos do sistema são explorados a partir de vários pontos de vista, contribuindo com informações para o processo de engenharia de requisitos. A diversidade dos envolvidos pode levar a requisitos emergentes que são inconsistentes ou conflitantes, tornando essencial a classificação dessas informações para facilitar a seleção de um conjunto coerente de requisitos internos. A obtenção de requisitos satisfatórios para o software pode apresentar vários desafios, como metas de projeto imprecisas, diferentes prioridades entre os envolvidos e suposições não explicitadas. O objetivo da engenharia de requisitos eficaz é eliminar ou reduzir esses problemas. Os engenheiros de requisitos têm o papel de identificar as áreas em comum (requisitos com os quais todos os envolvidos concordam) e de conflito ou inconsistência (requisitos desejados por um envolvido, mas que estão em conflito com os de outro envolvido). A tomada de decisão final sobre quais requisitos serão considerados e cortados fica a cargo do "defensor dos projetos", que pode ser, por exemplo, um gerente comercial ou um técnico sênior. Na concepção do projeto, as perguntas devem ser "livres de contexto", focando nos envolvidos, benefícios e metas globais. Elas ajudam a identificar os interessados, os benefícios mensuráveis e as possíveis alternativas para o desenvolvimento do software. Outro conjunto de perguntas visa compreender melhor o problema e permite que o cliente expresse suas percepções sobre uma solução. Por fim, um conjunto de "metaperguntas" concentra-se na eficiência da comunicação. Embora uma abordagem de perguntas e respostas seja útil no início, ela deve ser substituída posteriormente por uma abordagem mais abrangente que combine resolução de problemas, negociação e especificação de requisitos.

Requisitos não funcionais

Requisitos não funcionais (NFRs) são atributos de qualidade, desempenho, segurança ou restrições gerais em um sistema. Eles podem ser difíceis de articular pelos envolvidos. A proposta de Chung sugere uma abordagem em duas fases, onde na primeira fase são estabelecidas diretrizes de engenharia de software, como estilo arquitetural e padrões de projeto, e em seguida listados os NFRs. Na segunda fase, os NFRs são priorizados e agrupados usando regras para decidir quais diretrizes implementar ou rejeitar.

Rastreabilidade

A rastreabilidade na engenharia de software refere-se ao processo documentado de mapeamento entre diferentes artefatos, como requisitos e casos de teste. Uma matriz de rastreabilidade é essencial para representar as relações entre esses artefatos, ajudando nas atividades de desenvolvimento e garantindo que todos os requisitos sejam considerados. No entanto, atualizar e manter essa matriz pode se tornar desafiador com o aumento do número de requisitos e artefatos.

Levantamento de requisitos

O levantamento de requisitos tem o objetivo de combina solução de problemas, elaboração, negociação e especificação, estimulando abordagem colaborativa e orientada a equipes. Os envolvidos trabalham juntos para identificar o problema, propor elementos da solução, negociar abordagens e especificar requisitos preliminares da solução.

Coleta colaborativa de requisitos

A coleta colaborativa de requisitos envolve a realização de reuniões com a participação de engenheiros de software e outros envolvidos, seguindo diretrizes básicas para facilitar o processo. Isso inclui estabelecer regras de preparação e participação, utilizar uma pauta formal, mas flexível, e ter um facilitador para conduzir a reunião. Um mecanismo de definições, como planilhas ou fóruns virtuais, é utilizado para identificar problemas, propor soluções e especificar requisitos preliminares. Antes da reunião, é gerado um pedido de produto distribuído a todos os participantes. Durante a reunião, os participantes criam listas de objetos, serviços, restrições e critérios de desempenho relacionados ao sistema. Essas listas são combinadas e discutidas para criar uma lista consensual que reflita o produto ou sistema a ser desenvolvido. Mini-especificações são desenvolvidas para detalhar as entradas nas listas, e questões pendentes são registradas para serem abordadas posteriormente. O objetivo é alcançar um entendimento compartilhado e uma lista de requisitos aceitáveis para todos os envolvidos.

Cenários de uso

Conforme os requisitos são coletados, já é possível ter uma visão geral das funções e características do produto. Isso é fundamental para entender as funções e características do sistema a ser desenvolvido. A coleta de requisitos é o primeiro passo essencial no desenvolvimento de software, pois ajuda a formar uma visão clara do que o sistema precisa realizar e como ele será utilizado pelos diferentes tipos de usuários.

Artefatos do levantamento de requisitos

Durante o levantamento de requisitos, diversos artefatos são produzidos, variando conforme o tamanho do sistema ou produto a ser construído. Para sistemas grandes, esses artefatos incluem: (1) uma declaração de necessidade e viabilidade; (2) uma declaração de escopo do sistema ou produto; (3) uma lista dos participantes do levantamento de requisitos; (4) uma descrição do ambiente técnico do sistema; (5) uma lista de requisitos organizados por função e as restrições do domínio aplicáveis a cada um; e (6) um conjunto de cenários de utilização para ilustrar o uso do sistema em diferentes condições. Todos esses artefatos são revisados por todas as partes envolvidas no levantamento de requisitos.



Ferramentas Empregadas:

A elaboração de casos de uso é essencial para descrever a interação de um usuário com um sistema em situações específicas. Um caso de uso pode ser uma narrativa, uma descrição de tarefas ou uma representação esquemática que mostra o sistema do ponto de vista do usuário. O primeiro passo é identificar os atores envolvidos, que representam diferentes entidades que interagem com o sistema e têm objetivos específicos. Os atores não são necessariamente os mesmos que os usuários, já que um usuário pode desempenhar vários papéis, enquanto um ator representa uma classe de entidades externas que desempenham um único papel. Durante a coleta de requisitos, os atores podem ser classificados como primários ou secundários, dependendo de sua interação direta com o sistema e do suporte fornecido aos atores primários. Após a identificação dos atores, os casos de uso podem ser elaborados para descrever as interações entre eles e o sistema.

O objetivo do modelo de análise em engenharia de requisitos é descrever os domínios informacional, funcional e comportamental necessários para um sistema de computador. Ele é dinamicamente ajustado à medida que mais informações são adquiridas sobre o sistema e as necessidades dos stakeholders se tornam mais claras. O modelo pode evoluir ao longo do tempo, com alguns elementos se tornando estáveis e outros mais voláteis, refletindo o entendimento em desenvolvimento dos requisitos.

Existem diversas maneiras de representar os requisitos, e muitos profissionais defendem a utilização de múltiplas abordagens para contemplar diferentes perspectivas, o que auxilia na identificação de omissões e inconsistências.

Elementos baseados em cenários:

esses elementos comuns incluem cenários de uso, baseados nas interações do usuário com o sistema, e classes, que categorizam objetos manipulados pelo sistema.

Elementos baseados em classes:

Classes são elementos baseados em cenários de uso, onde um ator interage com o sistema por meio de objetos categorizados em classes. Uma maneira de identificar classes é procurar substantivos descritivos em um texto de caso de uso, e os verbos podem ser considerados métodos candidatos para as classes.

Elementos comportamentais:

como diagramas de estados, que descreve o comportamento do sistema em resposta a eventos podendo ser observável as mudanças que ocorrem.

Padrões de análise são soluções frequentemente recorrentes em domínios específicos, integrados ao modelo de análise para reutilização. Eles são armazenados em repositórios para facilitar sua busca e aplicação em projetos futuros.



Métricas:

No mundo ideal, as tarefas de engenharia de requisitos definem os detalhes necessários para prosseguir com o desenvolvimento de software. No entanto, na prática, é comum iniciar negociações com os envolvidos para equilibrar funcionalidade, desempenho e custo. O objetivo é desenvolver um plano que atenda às necessidades de todos e reflita as restrições reais, como tempo e orçamento. Fricker e colegas sugerem substituir a transferência tradicional de especificações por um processo de comunicação bidirecional chamado handshake, que promove um resultado ganha-ganha. Nesse processo, a equipe de software propõe soluções para os requisitos, descreve seu impacto e comunica seus objetivos para os representantes do cliente, que examinam as soluções propostas e buscam esclarecimentos. O handshake tende a melhorar a identificação, análise e escolha de variantes, promovendo uma negociação ganha-ganha.

O desenvolvimento incremental é comum na evolução de software, com contínua integração do código-fonte e desenvolvimento de novos casos de teste. O monitoramento de requisitos é útil nesse contexto, abrangendo tarefas como depuração distribuída, verificação em tempo de execução, validação das metas do usuário, monitoramento da atividade comercial e evolução colaborativa. Isso implica na necessidade de validação incremental, com suporte à validação contínua por meio da análise de modelos de meta do usuário em relação ao sistema em uso.

Conforme os elementos do modelo de requisitos são criados, eles são examinados em relação a inconsistências, omissões e ambiguidades. Isso se aplica mesmo aos modelos de processo ágil, nos quais os requisitos são frequentemente elaborados na forma de histórias de usuário e/ou casos de teste. Os requisitos são priorizados pelos envolvidos e agrupados em pacotes que serão implementados como incrementos de software. Uma revisão do modelo de requisitos aborda diversas questões para garantir que ele reflita de maneira precisa as necessidades do envolvido e forneça uma base sólida para o projeto.



Conclusão:

A engenharia de requisitos é o processo extremamente importante para estabelecer uma base sólida no desenvolvimento de software. Ela ocorre durante a comunicação com o cliente e as atividades de modelagem, abrangendo sete atividades: concepção, levantamento, elaboração, negociação, especificação, validação e gestão. No estágio de concepção, os requisitos básicos são definidos, juntamente com as restrições e características principais do sistema. O levantamento refina esses requisitos através de reuniões e cenários de uso. A elaboração expande os requisitos em um modelo, fazendo referência a padrões de análise. A negociação é realizada para determinar prioridades, disponibilidade e custos relativos dos requisitos. Todos os requisitos são validados em relação às necessidades do cliente para garantir a construção do sistema correto.
